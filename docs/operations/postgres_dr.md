# Postgres Disaster Recovery Runbook

Last-updated: 2026-04-19 · First line of defense against accidental `DROP
TABLE`, bad migration, silent corruption, or disk failure.

## TL;DR

- Nightly `pg_dump -Fc` runs at **22:00 UTC (05:00 Jakarta)** via Windows
  Scheduled Task `CortexDBBackup`
- Dumps land in `g:\AI_Trading_Bot\db_backups\cortex-YYYY-MM-DD.dump`
- Retention: last **7 daily + 4 weekly (Sundays) + 3 monthly (1st)**
- `engineered_features` + `feature_vectors` are **excluded** — they're
  derived from `ohlcv_bars` + model artifacts and regenerated by the
  monthly retrain job, so backing them up would 50× the bundle size
  with zero recovery value
- **This covers the DB only.** Full-system DR (including `.env` secrets
  + model artifacts + Scheduled Tasks) is tracked separately under
  **DR-1** in [BACKLOG.md](./BACKLOG.md)

## What gets backed up

| Table | Size | Why it matters |
|---|---:|---|
| `trades` | small | realized trade history, PnL |
| `signals` | small | every signal attempt, incl. `executed` flag |
| `equity_history` | small | 5-min equity snapshots (drawdown audit trail) |
| `execution_events` | small | order fill audit (R-1b slippage analysis) |
| `backtest_runs` | small | backtest run metadata |
| `backtest_trades` | small | per-trade backtest outcomes |
| `backtest_equity` | 276 MB | per-bar equity for backtest drill-down |
| `ohlcv_bars` | 238 MB | MT5 bar cache; re-fetchable but slow |
| `model_versions` | small | retrain audit trail |
| `model_predictions` | small | rolling LSTM prediction history |
| `actual_outcomes` | small | post-trade labels (for feedback loop) |

Excluded (regenerable on monthly retrain):
- `engineered_features` (~8.6 GB)
- `feature_vectors` (~1 GB)

Typical dump size after `-Fc` compression: **~20-30 MB**.

## Daily backup mechanics

Script: [`scripts/db_backup.ps1`](../scripts/db_backup.ps1)
Scheduled-Task installer: [`scripts/install_db_backup_task.ps1`](../scripts/install_db_backup_task.ps1)
Log file: `data/logs/db_backup.log` (tailed by the script, never rotated)

To run manually:

```powershell
powershell.exe -ExecutionPolicy Bypass -File scripts\db_backup.ps1
# or via the Task Scheduler:
Start-ScheduledTask -TaskName 'CortexDBBackup'
```

Dry-run (no files written, shows what would happen):

```powershell
powershell.exe -ExecutionPolicy Bypass -File scripts\db_backup.ps1 -DryRun
```

## Restore scenarios

### Scenario A · Restore to scratch DB for forensics (safe)

Use when investigating a bug or verifying that a backup is readable,
without touching production data.

```powershell
# 1. Create empty scratch DB
$env:PGPASSWORD = "<cortex_pass_from_.env>"
$psql = "C:\Program Files\PostgreSQL\18\bin\psql.exe"
& $psql -h localhost -p 5432 -U cortex -d postgres -c "DROP DATABASE IF EXISTS cortex_restore_test"
& $psql -h localhost -p 5432 -U cortex -d postgres -c "CREATE DATABASE cortex_restore_test"

# 2. Restore latest dump into it
$restore = "C:\Program Files\PostgreSQL\18\bin\pg_restore.exe"
$latest = Get-ChildItem g:\AI_Trading_Bot\db_backups\cortex-*.dump |
          Sort LastWriteTime -Descending | Select -First 1
& $restore -h localhost -p 5432 -U cortex -d cortex_restore_test `
           --no-owner --no-acl $latest.FullName

# 3. Query the scratch DB — connect string:
#    postgresql://cortex:<pass>@localhost:5432/cortex_restore_test

# 4. When done, drop it
& $psql -h localhost -p 5432 -U cortex -d postgres -c "DROP DATABASE cortex_restore_test"
Remove-Item Env:\PGPASSWORD
```

### Scenario B · Restore a specific table from a dump (surgical)

Use when ONE table got corrupted or wiped (e.g., accidental `DELETE FROM
trades`). Restore only that table into production without touching
anything else.

```powershell
$env:PGPASSWORD = "<cortex_pass_from_.env>"
$restore = "C:\Program Files\PostgreSQL\18\bin\pg_restore.exe"

# 1. STOP the bot so nothing writes while we restore
Get-CimInstance Win32_Process | Where-Object { $_.CommandLine -match 'main\.py' } |
    ForEach-Object { taskkill /F /T /PID $_.ProcessId }

# 2. Restore just the `trades` table (drops + recreates + loads data)
& $restore -h localhost -p 5432 -U cortex -d trading_bot `
           --no-owner --no-acl `
           --clean --if-exists `
           -t trades `
           g:\AI_Trading_Bot\db_backups\cortex-2026-04-19.dump

# 3. Restart the bot
Start-ScheduledTask -TaskName 'CortexTradingBot'
Remove-Item Env:\PGPASSWORD
```

Swap `-t trades` for the table(s) you need. Multiple `-t` flags are
additive.

### Scenario C · Full restore after catastrophic loss (nuclear)

Use when the entire `trading_bot` database is gone (`DROP DATABASE`
accident, cluster-level corruption, etc.).

```powershell
$env:PGPASSWORD = "<cortex_pass_from_.env>"
$psql = "C:\Program Files\PostgreSQL\18\bin\psql.exe"
$restore = "C:\Program Files\PostgreSQL\18\bin\pg_restore.exe"

# 1. STOP the bot
Get-CimInstance Win32_Process | Where-Object { $_.CommandLine -match 'main\.py' } |
    ForEach-Object { taskkill /F /T /PID $_.ProcessId }

# 2. Drop the broken DB (make absolutely sure no one else is on it first!)
& $psql -h localhost -p 5432 -U cortex -d postgres -c `
  "SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE datname='trading_bot'"
& $psql -h localhost -p 5432 -U cortex -d postgres -c "DROP DATABASE IF EXISTS trading_bot"
& $psql -h localhost -p 5432 -U cortex -d postgres -c "CREATE DATABASE trading_bot"

# 3. Restore the latest dump
$latest = Get-ChildItem g:\AI_Trading_Bot\db_backups\cortex-*.dump |
          Sort LastWriteTime -Descending | Select -First 1
Write-Host "Restoring from $($latest.Name)"
& $restore -h localhost -p 5432 -U cortex -d trading_bot `
           --no-owner --no-acl --clean --if-exists `
           $latest.FullName

# 4. Regenerate the derived tables (excluded from dumps). The monthly
#    retrain normally does this; running it manually takes ~30 min but
#    isn't strictly necessary for trading to resume — the bot will
#    repopulate them on the next H4 cycle.
#    If you want them now, run:
#        venv\Scripts\python.exe scripts\train_deep_learning.py --symbols XAUUSD EURUSD USDJPY USDCAD ETHUSD --triple-barrier --softmax

# 5. Restart the bot
Start-ScheduledTask -TaskName 'CortexTradingBot'
Remove-Item Env:\PGPASSWORD
```

Data loss: whatever was written between the last backup and the failure.
With 22:00 UTC daily backups, **worst case ≈ 24 hours** of trades /
signals / equity snapshots.

### Scenario D · Point-in-time roll-back (older backup)

Use when corruption went unnoticed for a day and the latest dump already
has the bad data. List available dumps and pick the one from BEFORE the
issue:

```powershell
Get-ChildItem g:\AI_Trading_Bot\db_backups\cortex-*.dump |
    Sort LastWriteTime -Descending |
    Format-Table Name, LastWriteTime, @{N='SizeMB';E={[math]::Round($_.Length/1MB,1)}}
```

Then run Scenario C substituting `$latest` with the specific dump you
picked.

## Monitoring

The backup script always writes to `data/logs/db_backup.log`. Tail it
to confirm recent runs:

```powershell
Get-Content g:\AI_Trading_Bot\Cortex\data\logs\db_backup.log -Tail 30
```

Scheduled-Task status:

```powershell
Get-ScheduledTask -TaskName 'CortexDBBackup' | Get-ScheduledTaskInfo |
    Format-List LastRunTime, LastTaskResult, NextRunTime
```

`LastTaskResult = 0` means success. Anything else means investigate the
log. `LastTaskResult = 267011` is "task has not yet run" — normal right
after install.

Future work (tracked in [BACKLOG.md](./BACKLOG.md) DR-1):
a dashboard indicator that flags when `db_backup.log`'s most-recent entry
is > 26 hours old, so silent scheduler failures surface within a day.

## Gotchas

- **Password in env var.** `db_backup.ps1` reads the password from `.env`
  and sets `$env:PGPASSWORD` for the duration of the dump, then unsets
  it. Don't let anyone run `ps -e` during the narrow window between set
  and unset — it's already not in the argv list so this is mostly a
  belt-and-suspenders concern.
- **DB size growing.** If dump size > 100 MB sustained, check whether
  `backtest_equity` has accumulated many stale runs. `DELETE FROM
  backtest_equity WHERE run_id NOT IN (SELECT id FROM backtest_runs
  WHERE status = 'done' ORDER BY created_at DESC LIMIT 50)` prunes old
  backtest noise.
- **Retention gap on weekly/monthly.** The weekly slot requires a dump
  ON Sunday; the monthly slot requires ON the 1st. If the machine was
  off on those days, the slot stays empty until the next matching date.
  This is by design — no synthetic dumps — but means the "14-slot"
  retention can be less than 14 in practice.
- **Restore resets sequences.** `pg_restore` restores sequences to the
  values they had at dump time. If you restore an older backup and then
  write new rows, you might get duplicate-key errors until sequences
  catch up. Run `SELECT setval(pg_get_serial_sequence('<table>','id'),
  (SELECT MAX(id) FROM <table>))` for any table with duplicate-key
  errors after a restore.
